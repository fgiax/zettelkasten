# Гайд по веткам в Git

Гайд по веткам в Git — без лишнего шума, только то, что реально нужно на практике.


---

🌿 Git ветки: создание, переключение, работа

🧠 1. Что такое ветка

Ветка — это просто отдельная линия разработки.

main — основная версия проекта

for example: `try_ui` — твоя экспериментальная версия


👉 Они независимы: изменения в одной не влияют на другую, пока ты их не сольёшь (merge).


---

🌱 2. Посмотреть текущие ветки

git branch

Пример вывода:

* main
  try_ui

Звёздочка * — где ты сейчас находишься.


---

➕ 3. Создать и переключиться на новую ветку

Современный способ (рекомендуется):

git switch -c try_ui

👉 делает сразу 2 вещи:

- создаёт ветку try_ui

- переключает тебя на неё



---

🔁 4. Переключиться на существующую ветку

git switch main

или

git switch try_ui


---

✏️ 5. Работа в ветке (обычный цикл)

После переключения:

git add .
git commit -m "my UI experiment"

👉 всё это сохраняется только в текущей ветке


---

🔄 6. Вернуться обратно

git switch main


---

⚠️ 7. Если есть незакоммиченные изменения

Git может не дать переключиться.

Вариант 1 — сохранить через commit:

git add .
git commit -m "save work"
git switch main

Вариант 2 — временно спрятать:

git stash
git switch main

Вернуть обратно:

git stash pop


---

🗑️ 8. Удалить ветку (если больше не нужна)

git branch -d try_ui

(если Git ругается — значит изменения не слиты)


---

⚔️ 9. switch vs checkout (важно понимать)

Команда	Использование

git switch	только работа с ветками (рекомендуется)
git checkout	старый универсальный способ (ветки + файлы)


Примеры:

git switch try_ui
git switch main
git switch -c new_branch


---

🧭 10. Мини-шпаргалка

git branch                # посмотреть ветки
git switch -c try_ui     # создать и перейти
git switch main          # вернуться
git branch -d try_ui     # удалить ветку
git stash                # спрятать изменения
git stash pop            # вернуть изменения


---

🚀 Рекомендуемый workflow (как делают на практике)

1. Работаешь в main


2. Создаёшь ветку:

git switch -c feature_ui


3. Экспериментируешь / пишешь код


4. Коммитишь


5. Возвращаешься в main


6. Потом уже делаешь merge (если нужно)




---

👉 как правильно делать merge без конфликтов
👉 или как выглядит workflow как в реальных Android-командах (feature branches + pull request)
